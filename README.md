# 🍼 SporraMom W3 智能客服系统

基于 RAG（检索增强生成）的智能问答系统，为 SporraMom W3 穿戴式吸奶器提供 7×24 小时自动客服。

## 技术架构

```
用户问题
    ↓
┌─────────────────────────────────────────────┐
│  意图过滤 (LLM)                              │
│  闲聊/超范围 → 直接回复                       │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  混合检索                                     │
│  ┌──────────┐  ┌──────────┐                  │
│  │ BM25     │  │ 向量检索  │                  │
│  │ 关键词   │  │ 语义理解  │                  │
│  └────┬─────┘  └────┬─────┘                  │
│       └──────┬──────┘                        │
│         RRF 融合 (Reciprocal Rank Fusion)     │
│              ↓                               │
│        Reranker 精排 (CrossEncoder)           │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  相关性检查                                    │
│  分数 < 阈值 → 降级策略 (扩大检索/转人工)      │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│  答案生成 (mimo-v2.5-pro)                     │
│              ↓                               │
│  自检: 关键词覆盖率检查                        │
│  覆盖率低 → 低温度重试                         │
└─────────────────┬───────────────────────────┘
                  ↓
            最终答案 + 引用来源
```

## 核心模块

| 文件 | 功能 | 面试展示点 |
|------|------|-----------|
| `data_processor.py` | 数据处理: Markdown 解析→语义切分→元数据 | 数据收集与处理策略 |
| `retriever.py` | 混合检索: BM25 + 向量 + RRF + Rerank | 检索技术选型与融合策略 |
| `rag_pipeline.py` | RAG 主流程: 意图过滤→检索→生成→自检 | 工作流稳定性设计 |
| `evaluator.py` | 评测体系: 测试集 + 多维指标 | 模型评测与迭代方法 |
| `app.py` | Gradio 界面: 聊天 + 可视化 + 评测 | 产品化展示 |
| `config.py` | 配置管理: 所有可调参数 | A/B 实验支持 |

## 面试准备: 三个核心问题

### 1. 数据收集与处理

**问题**: "你们是如何收集、准备和处理数据的？"

**回答要点**:
- **数据来源**: 产品说明书 (结构化 Markdown)，是产品最权威的信息源
- **切分策略**: 按 Markdown 标题层级切分，而非固定 token 切分
  - 原因: 固定 token 切分会割裂语义，比如把"安全警告"切成两半
  - 按章节切分保证每个 chunk 是一个完整的知识点
- **元数据**: 每个 chunk 带 section_path (章节路径)，用于追溯来源
- **超长段落**: 二次切分 + overlap，避免信息在切分边界丢失
- **缓存机制**: 基于文件 hash 的缓存，文件变了自动重建

**可以展示**: 运行 `data_processor.py`，展示 chunks 的统计信息和切分效果。

### 2. 模型评测体系

**问题**: "如何证明你的模型效果变好了？"

**回答要点**:
> "AI 产品经理必须有一把评测的尺子。我们的尺子由三层指标构成:"

**第一层: 检索质量** (衡量"找没找对")
- **Hit Rate@K**: top-K 结果中是否包含正确文档
  - 业务含义: 用户问的问题，系统能不能"找到"相关文档
- **MRR**: 正确文档排在第几位
  - 业务含义: 相关文档排名越靠前，LLM 越容易生成好答案

**第二层: 生成质量** (衡量"答没答好")
- **关键词覆盖率**: 答案是否包含期望的关键信息
- **LLM 相关性评分**: 用 LLM 评判答案与问题的相关性 (1-5分)

**第三层: 系统指标** (衡量"稳不稳定")
- **响应时间**: 用户等待体验
- **降级触发率**: 系统"答不上来"的比例
- **自检通过率**: 答案质量自检通过的比例

**如何迭代**:
> "每次调整一个变量，用同一测试集对比:
> - 调整 chunk_size → 对比 Hit Rate (切分更精准？)
> - A/B: 纯 BM25 vs 混合检索 → 对比 MRR (语义检索有帮助？)
> - 调整 temperature → 对比关键词覆盖率 (更忠实于原文？)"

**可以展示**: 运行评测面板，展示 25 个测试用例的结果和失败分析。

### 3. 工作流稳定性

**问题**: "如何保证工作流的效率、稳定性和容错性？"

**回答要点**:
> "我在流程中设计了三道防线:"

**防线1: 意图过滤**
- 用户问闲聊/超出范围 → 直接礼貌回复，不走检索
- 避免无关问题污染检索结果，节省计算资源

**防线2: 相关性阈值检查**
- Rerank 最高分 < 阈值 → 说明知识库中没有相关内容
- 降级策略链:
  1. 扩大检索范围 (top_k × 2)
  2. 放宽阈值重试 (× 0.8)
  3. 仍不足 → 转人工客服
- 避免"硬答"导致幻觉

**防线3: 答案自检**
- 生成的答案与源文档做关键词覆盖率检查
- 覆盖率低 → 用更低温度重新生成 (提高忠实度)
- 避免模型编造不在文档中的信息

**可以展示**: 检索可视化面板，展示每一步的中间结果和决策过程。

## 扩展方向

1. **Query Expansion**: 用户说"充不了电"→ 扩展为"充电 无法充电 电池 适配器"
2. **HyDE**: 先让 LLM 生成假设性答案，用假设答案做向量检索
3. **多轮对话**: 记忆上下文，支持追问
4. **用户反馈**: 收集"有帮助/没帮助"，用于持续优化
5. **知识库扩展**: 接入更多产品文档、FAQ、客服历史记录

## 技术栈

- **LLM**: mimo-v2.5-pro (Anthropic 兼容 API)
- **Embedding**: BAAI/bge-small-zh-v1.5 (~90MB, 中文优化)
- **Reranker**: BAAI/bge-reranker-base (CrossEncoder)
- **BM25**: rank_bm25 (纯 Python)
- **向量库**: FAISS (本地, 无需服务)
- **UI**: Gradio 4.x/6.x
