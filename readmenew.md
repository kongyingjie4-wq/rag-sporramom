# SporraMom W3 智能客服系统 — 产品文档

## 一、项目背景与问题定义

### 1.1 业务痛点

SporraMom W3 是一款穿戴式吸奶器，目标用户是哺乳期妈妈。这类用户有三个显著特征：

- **时间碎片化**: 常在夜间、哺乳间隙使用产品，无法等待人工客服
- **情绪敏感**: 哺乳期妈妈对疼痛、安全问题高度焦虑，需要即时、准确的回答
- **使用场景紧急**: 吸奶器故障（如无法开机、吸力异常）直接影响哺乳计划

**核心矛盾**: 用户需要 7×24 小时即时响应，但人工客服无法覆盖全时段。

### 1.2 产品目标

| 目标 | 衡量标准 | 优先级 |
|------|----------|--------|
| 降低人工客服压力 | 常见问题自动解决率 ≥ 70% | P0 |
| 提升用户满意度 | 回答准确率 ≥ 85% | P0 |
| 保障安全信息传达 | 安全类问题回答准确率 100% | P0 |
| 控制回答风险 | 幻觉率（编造信息）< 5% | P0 |
| 快速响应 | 平均响应时间 < 20s | P1 |

## 三、产品方案设计

### 3.1 整体架构

采用 **RAG（检索增强生成）** 架构，核心逻辑是：

> **先从知识库中找到相关文档，再基于文档生成回答。**

这样做的好处是：回答有据可查，不是凭空编造。

```
用户提问
    ↓
  理解意图（是产品问题？闲聊？超出范围？）
    ↓
  从知识库中检索相关内容（关键词 + 语义双重匹配）
    ↓
  判断检索结果是否足够回答问题
    ↓ 不够 → 降级：引导联系人工客服
  基于检索内容生成回答
    ↓
  自检：回答是否忠于原文
    ↓ 不通过 → 重新生成
  返回回答 + 引用来源
```

### 3.2 关键设计决策

#### 决策1: 为什么用"混合检索"而非单一检索？

| 方案 | 优势 | 劣势 |
|------|------|------|
| 纯关键词检索 (BM25) | 精确匹配型号、参数 | 理解不了同义词（"充不了电" ≠ "无法充电"） |
| 纯语义检索 (向量) | 理解用户意图 | 对精确参数匹配弱（"1400mAh"） |
| **混合检索 + 重排序** | **兼顾两者** | 实现复杂度略高 |

**产品决策**: 用混合检索，关键词和语义各取所长，通过 RRF 融合排序，再用精排模型做最终筛选。

**业务收益**: 用户说"充不了电"，语义检索能理解这是"充电故障"；用户说"IP22"，关键词检索能精确匹配技术参数。

#### 决策2: 为什么设计"三道防线"？

作为产品经理，我最担心的是**模型编造信息（幻觉）**，尤其是在安全相关问题上。

| 防线 | 触发条件 | 处理方式 | 防什么 |
|------|----------|----------|--------|
| 意图过滤 | 用户问的是闲聊/超范围 | 直接回复，不走检索 | 无关问题浪费资源 |
| 相关性阈值 | 检索结果与问题不匹配 | 降级策略：扩大检索→放宽阈值→转人工 | 硬答导致幻觉 |
| 答案自检 | 答案关键词覆盖率低 | 低温度重试（更忠实原文） | 编造不在文档中的信息 |

**产品原则**: 宁可说"我不知道，请联系客服"，也不能编造一个看似合理但错误的答案。对于吸奶器这类涉及母婴安全的产品，错误信息的代价远大于"没回答"。

#### 决策3: 为什么按"章节"而非"固定长度"切分文档？

| 方案 | 效果 |
|------|------|
| 固定 500 字切分 | "安全警告"可能被切成两半，检索到一半信息不完整 |
| **按 Markdown 标题切分** | 每个 chunk 是一个完整知识点，检索到就能完整回答 |

**业务收益**: 用户问"充电时能用吗"，系统能检索到完整的"充电时禁止佩戴使用"安全警告，而不是只检索到半句话。

## 四、评测体系设计

### 4.1 评测理念

> "AI 产品经理必须有一把评测的尺子。没有量化指标，就无法证明效果变好了。"

评测体系分三层，从技术到业务逐层递进：

### 4.2 第一层：检索质量（技术指标）

| 指标 | 定义 | 业务含义 | 当前基线 |
|------|------|----------|----------|
| Hit Rate@3 | 前3个结果中是否包含正确文档 | 用户的问题能不能被"找到" | 72% |
| MRR | 正确文档排在第几位的倒数 | 相关文档排得越前面，答案越好 | 0.553 |

**迭代方法**: 调整 chunk 策略、检索参数 → 重新评测 → 对比 Hit Rate 变化。

### 4.3 第二层：生成质量（业务指标）

| 指标 | 定义 | 业务含义 | 当前基线 |
|------|------|----------|----------|
| 关键词覆盖率 | 答案中包含期望关键词的比例 | 关键信息有没有遗漏 | 81.4% |
| LLM 相关性评分 | LLM 评判答案与问题的相关性 (1-5分) | 有没有答非所问 | 可选开启 |

**迭代方法**: 调整 prompt、temperature → 重新评测 → 对比覆盖率。

### 4.4 第三层：系统指标（产品指标）

| 指标 | 定义 | 业务含义 | 当前基线 |
|------|------|----------|----------|
| 响应时间 | 用户等待时长 | 用户体验 | 17.6s |
| 降级触发率 | "答不上来"的比例 | 知识库覆盖度 | 8% |
| 自检通过率 | 答案质量自检通过比例 | 回答可信度 | 100% |

### 4.5 测试集设计

测试集共 **25 个用例**，覆盖 6 大类场景：

| 类别 | 用例数 | 设计原则 |
|------|--------|----------|
| 安全警告 | 5 | 必须 100% 正确，零容忍 |
| 操作步骤 | 3 | 步骤完整性 |
| 故障排除 | 4 | 现象→原因→对策的完整链路 |
| 技术参数 | 4 | 精确数值匹配 |
| 尺寸/清洁/存储 | 5 | 常见使用问题 |
| 边界/陷阱 | 4 | 超范围、对比类问题 |

**设计原则**: 测试集不是随便写几个问题，而是要覆盖真实用户的高频场景和容易出错的边界情况。

## 五、风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 模型幻觉（编造信息） | 用户被误导，安全风险 | 三道防线 + 安全问题零容忍 |
| 知识库覆盖不全 | 用户问题答不上来 | 降级策略 + 引导人工客服 |
| 检索不到相关内容 | 答案偏离主题 | 相关性阈值 + 扩大检索范围 |
| 响应时间过长 | 用户流失 | 轻量模型 + 缓存机制 |

## 六、迭代路线图

### Phase 1: MVP（当前）
- 基于产品说明书的单轮问答
- 混合检索 + Rerank
- 基础评测体系

- 多轮对话（记忆上下文）

- 接入 FAQ 库、客服历史工单

- 主动推送（根据用户使用阶段推荐内容）
- 情绪识别（焦虑情绪优先安抚）
- 工单自动创建（无法解决时自动转人工并附带上下文）

## 七、数据处理说明

### 7.1 数据来源

| 数据类型 | 来源 | 说明 |
|----------|------|------|
| 产品说明书 | 官方 Markdown 文档 | 权威信息源，涵盖安全/操作/参数/故障 |
| FAQ | (规划中) | 客服高频问题汇总 |
| 工单记录 | (规划中) | 真实用户问题和解决方案 |

### 7.2 数据处理流程

```
原始 Markdown 文档
    ↓
按标题层级解析（保留章节结构）
    ↓
语义切分（每个 chunk 是一个完整知识点）
    ↓
添加元数据（章节路径、chunk ID、字符数）
    ↓
缓存（基于文件 hash，文件变了自动重建）
```

### 7.3 切分策略对比

| 策略 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 固定 token 切分 | 实现简单 | 割裂语义，安全警告可能被切两半 | ❌ |
| 按句子切分 | 语义完整 | 太碎，缺乏上下文 | ❌ |
| **按 Markdown 标题切分** | **语义完整，保留层级** | 依赖文档结构 | ✅ |
| 超长段落二次切分 | 处理长章节 | 需要 overlap 机制 | ✅ (补充) |

## 八、系统架构说明

### 8.1 技术选型理由

| 组件 | 选型 | 选择理由 |
|------|------|----------|
| LLM | mimo-v2.5-pro | 中文能力强，API 兼容 OpenAI 格式 |
| Embedding | bge-small-zh-v1.5 | 中文优化，仅 90MB，GPU 友好 |
| Reranker | bge-reranker-base | CrossEncoder 架构，比双塔更精准 |
| 关键词检索 | BM25Okapi | 无需训练，对精确匹配效果好 |
| 向量库 | FAISS | 本地部署，无需外部服务 |
| 前端 | Gradio | 快速搭建，内置聊天组件 |

### 8.2 检索流程详解

```
用户问题: "吸奶器充不了电怎么办"

Step 1: BM25 关键词检索
  → jieba 分词: [吸奶器, 充不了, 电, 怎么办]
  → 匹配文档: 安全警告(充电相关)、故障排除(无法充电)、电池&充电

Step 2: 向量语义检索
  → 编码为 512 维向量
  → 余弦相似度匹配: 故障排除(无法充电)、安全警告(充电时禁止)

Step 3: RRF 融合
  → 综合两个检索器的排名，不依赖分数绝对值
  → 融合后排序: 故障排除(无法充电) > 安全警告 > 电池&充电

Step 4: Reranker 精排
  → CrossEncoder 对 query-doc 对打分
  → 最终排序: 故障排除(无法充电) > 电池&充电 > 安全警告
```

### 8.3 容错机制

```
检索结果相关性 < 阈值？
    ↓ 是
扩大检索范围 (top_k × 2)
    ↓ 仍然不足
放宽阈值重试 (× 0.8)
    ↓ 仍然不足
返回降级回复: "建议联系客服 support@sporramom.com"
```

**设计原则**: 每一步都有可观测的日志和指标，出问题时能快速定位是检索环节还是生成环节。
