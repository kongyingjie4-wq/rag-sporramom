# SporraMom W3 智能客服系统 — 完整技术文档

## 一、系统架构

### 1.1 整体流程

```
用户提问
    ↓
Step 1: 闲聊过滤（规则，0ms）
    "你好/谢谢/拜拜" → 闲聊回复
    其他 → 继续
    ↓
Step 2: 混合检索（~11ms 无 Rerank / ~7.5s 有 Rerank）
    ├── BM25 关键词检索：jieba 分词 → BM25Okapi 打分
    ├── 向量语义检索：bge-small-zh 编码 → FAISS 余弦相似度
    └── RRF 融合：score = 1/(60+BM25排名) + 1/(60+向量排名)
    ↓
Step 3: Rerank 精排（可选，~7.5s）
    bge-reranker-base CrossEncoder 打分
    config.RERANK_ENABLED = True/False 切换
    ↓
Step 4: 相关性检查（0ms）
    最高分 >= 阈值？→ 继续
    最高分 < 阈值？→ 低置信度提示 / 转人工
    ↓
Step 5: LLM 生成答案（~6-11s，取决于 API）
    读取检索到的 chunk 内容 → mimo-v2.5-pro 生成回答
    ↓
Step 6: 自检（0ms）
    答案关键词是否在源文档中出现
    覆盖率 < 30% → 低温度重试
    ↓
返回最终答案
```

### 1.2 各步骤耗时分析

| 步骤 | 耗时 | 是否调用 LLM | 说明 |
|------|------|-------------|------|
| 闲聊过滤 | 0ms | 否 | 规则匹配 |
| BM25 检索 | ~1ms | 否 | jieba 分词 + BM25 打分 |
| 向量检索 | ~10ms | 否 | bge 编码 + FAISS 搜索 |
| RRF 融合 | <1ms | 否 | 数学计算 |
| Rerank 精排 | ~7.5s | 否 | CrossEncoder 打分（CPU 慢） |
| 相关性检查 | 0ms | 否 | 比较分数 |
| LLM 生成 | ~6-11s | 是 | API 调用（主要瓶颈） |
| 自检 | 0ms | 否 | 关键词匹配 |

**结论**：LLM 调用占总时间的 60-80%，是主要瓶颈。Reranker 是第二大瓶颈（7.5s），但可以通过配置关闭。

### 1.3 Reranker 开关对比

在 `config.py` 中设置 `RERANK_ENABLED = True/False`：

| 配置 | Hit Rate@3 | MRR | 关键词覆盖率 | 响应时间 |
|------|-----------|-----|------------|---------|
| RERANK_ENABLED = True | **92%** | **0.790** | **90.9%** | ~19s |
| RERANK_ENABLED = False | 76% | 0.690 | 84.7% | ~11s |

**面试展示点**：这是一个典型的质量 vs 速度的 trade-off，可以通过配置灵活切换。

## 二、数据处理：Chunk 切分策略

### 2.1 切分原则

**一个 chunk = 一个独立的知识点**

用户问一个问题，理想情况下只需要一个 chunk 就能回答。

### 2.2 V1 自动切分（失败案例）

最初用 Python 代码按 Markdown 标题自动切分：

```python
# 遇到 # ## ### 标题就切一刀
for line in lines:
    if line.startswith('##'):
        flush_previous_content()
        start_new_chunk()
```

**产生的问题：**

| 问题 | 具体表现 | 影响 |
|------|---------|------|
| chunk 太短 | Chunk 1 只有 21 字（产品名） | 向量编码质量差 |
| 无用 chunk | Chunk 5 是目录 | 浪费检索资源 |
| 长 chunk 没切 | 安全警告 12 条混在 769 字的大块里 | 检索不精准 |
| chunk 重叠 | 第 8-9 条警告重复 | 重复信息干扰排序 |
| 粒度不均 | 最短 21 字，最长 769 字 | 检索效果不稳定 |
| 缺口语化 | 只有书面语"分离主机"，没有口语"怎么拆" | 用户口语查询匹配差 |

### 2.3 V2 人工切分（优化后）

手动按逻辑单元切分，共 33 个 chunk：

| 类别 | chunk 数 | 切分逻辑 |
|------|---------|---------|
| 产品概述 | 1 | 合并标题+引言+适用范围 |
| 安全警告 | 12 | 每条警告独立，加"温馨提示" |
| 禁忌症 | 1 | 独立知识点 |
| 产品组成 | 1 | 6 个部件一起（有关联性） |
| 按钮操作 | 2 | 电源键/切换键 + 档位/释放按钮 |
| LED 显示 | 2 | 电量/充电 + 模式/档位 |
| 充电续航 | 1 | 充电时间、续航、电池容量合并 |
| 拆解步骤 | 1 | 加口语"怎么拆""拆卸""取下" |
| 尺寸选择 | 2 | 选择与后果 + 测量方法 |
| 清洁消毒 | 1 | 加口语"怎么清洗" |
| App 使用 | 1 | 加口语"怎么连接蓝牙" |
| 组装步骤 | 1 | 加口语"怎么装""安装" |
| 吸取母乳 | 1 | 加口语"怎么吸奶""使用方法" |
| 倒奶存奶 | 1 | 加口语"怎么保存" |
| 故障排除 | 3 | 充电/吸力/漏液各一个 |
| 技术规格 | 1 | 全部参数合并 |
| 保修服务 | 1 | 保修+售后联系方式 |

### 2.4 关键优化：口语化注释

**问题**：用户说"怎么拆"，chunk 里写"分离主机"——字不一样，BM25 匹配不上，向量分数也不够高。

**解决**：在 chunk 里加同义词注释。

```
改前: "按压两侧释放按钮，分离主机与集乳杯"
改后: "吸奶器怎么拆（拆解、拆卸、取下）：按压释放按钮，把主机分离"
```

加了同义词的 chunk：

| 专业词 | 口语化注释 |
|--------|----------|
| 硅胶隔膜 | 也叫硅胶片 |
| 喇叭罩 | 也叫吸乳罩、护罩 |
| 集乳杯 | 也叫奶杯 |
| 拆解 | 拆、拆卸、取下、卸下 |
| 组装 | 装、安装 |

**原理**：加同义词不是因为向量检索没用，而是让 BM25 字面匹配和向量语义匹配都变强，排名更靠前。

### 2.5 为什么不把按钮和 LED 每条拆开？

太短的 chunk（如"电源键，长按1.5秒开/关机"只有 15 字）向量编码质量差，信息太少语义模糊。而且用户问"怎么开机"时可能同时需要知道电源键和切换键。

**按功能分组**是最佳平衡：不太长不太短，语义完整。

### 2.6 为什么规格参数融在一个 chunk 没问题？

```
用户问: "噪音大吗"
chunk: "噪音<50dB，防护等级IP22，电池1400mAh..."

BM25: "噪音" 在 chunk 里出现 → 精确命中
向量: "噪音大吗" 和 "噪音<50dB" 语义接近 → 分数高
```

BM25 看关键词命中，向量看语义匹配。chunk 里规格越多，被命中的概率越高。LLM 能从十几行里找到对应的那一行。

## 三、混合检索原理

### 3.1 BM25 关键词检索

**作用**：精确匹配关键词，如型号"YM-8871"、参数"1400mAh"。

```
用户问题: "吸奶器 充不了 电"
         ↓ jieba 分词
关键词: [吸奶器, 充不了, 电]

遍历 33 个 chunk，计算 BM25 分数
→ 故障排除-充电问题 (分数最高)
→ 安全警告-充电时禁止使用
→ 充电时间与续航
```

### 3.2 向量语义检索

**作用**：理解用户意图，如"充不了电"匹配"无法充电"。

```
用户问题 → bge-small-zh 编码 → 512 维向量
33 个 chunk → 已有 33 个向量（启动时编码好存入 FAISS）

FAISS 计算余弦相似度
→ 故障排除-充电问题 (相似度 0.89)
→ 安全警告-充电时禁止使用 (相似度 0.82)
→ 充电时间与续航 (相似度 0.78)
```

### 3.3 RRF 融合

**作用**：合并 BM25 和向量两个排序，不依赖分数绝对值。

```
公式: score(doc) = 1/(60 + BM25排名) + 1/(60 + 向量排名)

示例:
  故障排除-充电问题: BM25 排第1 + 向量排第1
  → RRF = 1/61 + 1/61 = 0.0328

  安全警告-充电时禁止使用: BM25 排第2 + 向量排第2
  → RRF = 1/62 + 1/62 = 0.0323
```

**为什么用 RRF**：BM25 分数范围 0-10，向量相似度范围 0-1，量纲不同没法直接比较。RRF 只看排名，更鲁棒。

### 3.4 Reranker 精排（可选）

**作用**：对 RRF 的 top 候选做更精准的打分。

```
BM25/向量 = 双塔模型：问题和文档各自编码，独立比较
Reranker = CrossEncoder：问题和文档拼在一起输入，交叉注意力打分

双塔: [问题]→向量A ─┐
                     ├→ 比较
      [文档]→向量B ─┘

交叉: [问题+文档] → 一个模型 → 相关性分数
```

CrossEncoder 更准但更慢，所以只对 top-10 候选做。33 个 chunk 的知识库，RRF 已经够准，Reranker 是锦上添花。

## 四、评测体系

### 4.1 测试集设计

25 个测试用例，覆盖 6 大类场景：

| 类别 | 用例数 | 示例 |
|------|--------|------|
| 安全警告 | 5 | "充电时能用吗""孕期能用吗" |
| 操作步骤 | 3 | "怎么组装""按钮怎么用" |
| 故障排除 | 4 | "开不了机""没有吸力" |
| 技术参数 | 4 | "电池容量""噪音大吗" |
| 尺寸/清洁/存储 | 5 | "尺寸怎么选""母乳保存多久" |
| 边界/陷阱 | 4 | "和美德乐比""坏了自己修" |

每个用例标注：
- `query`: 用户问题
- `expected_keywords`: 答案应包含的关键词
- `expected_topic`: 应命中的 chunk 主题

### 4.2 指标计算

#### Hit Rate@3

```
25 个用例，每个检索后看 top-3 是否包含期望 chunk
Hit Rate@3 = 命中数 / 25
```

#### MRR（平均倒数排名）

```
正确 chunk 排第1 → 1/1 = 1.0
正确 chunk 排第2 → 1/2 = 0.5
正确 chunk 排第3 → 1/3 = 0.33
MRR = 所有用例的倒数排名取平均
```

#### 关键词覆盖率

```
期望关键词: ["充电", "禁止", "边充边用", "电气安全"]
答案包含: "充电" ✅ "禁止" ✅ "边充边用" ❌ "电气安全" ✅
覆盖率 = 3/4 = 75%
```

### 4.3 评测结果对比

| 指标 | V1（自动切分 23 个） | V2（人工切分 33 个） | 提升 |
|------|---------------------|---------------------|------|
| Hit Rate@3 | 72% | **92%** | +20% |
| MRR | 0.553 | **0.790** | +43% |
| 关键词覆盖率 | 81.4% | **90.9%** | +9.5% |
| 降级触发率 | 8% | **0%** | -8% |

分类别 Hit Rate@3：

| 类别 | V1 | V2 |
|------|-----|-----|
| 安全警告 | 100% | 100% |
| 操作步骤 | 67% | **100%** |
| 故障排除 | 50% | **100%** |
| 技术参数 | 25% | **75%** |

### 4.4 失败用例分析

| 问题 | 失败原因 | 优化方向 |
|------|---------|---------|
| "噪音大吗" | "噪音"在安全警告里出现（"静音表现"）干扰排序 | 在技术规格 chunk 加"噪音大""吵不吵" |
| "电池容量" | "充电时间与续航"和"技术规格"都有电池信息 | 考虑合并或加明确区分词 |
| "坏了自己修" | "修"和"拆解""清洁"语义接近 | 在保修 chunk 加"坏了""维修""自己修" |

## 五、容错机制

### 5.1 三道防线

```
防线1: 闲聊过滤
  "你好/谢谢" → 直接回复，不走检索
  避免无关问题浪费资源

防线2: 相关性阈值检查
  检索最高分 < 0.05 → 知识库中没有相关内容
  降级策略: 扩大检索 → 低置信度提示 → 转人工
  避免"硬答"导致幻觉

防线3: 答案自检
  关键词覆盖率 < 30% → 答案可能包含编造信息
  用 temperature=0 重新生成
  避免模型编造不在文档中的信息
```

### 5.2 设计原则

**宁可说"我不知道，请联系客服"，也不能编造一个看似合理但错误的答案。**

对于吸奶器这类涉及母婴安全的产品，错误信息的代价远大于"没回答"。

## 七、面试话术

### "你们是怎么做数据处理的？"

> "最初用代码按 Markdown 标题自动切分，但效果不好——chunk 粒度不均，安全警告 12 条混在 2 个大块里。
>
> 后来改成人工按逻辑单元切分，每个 chunk 一个独立知识点。安全警告每条独立，故障排除每个类型独立。同时在 chunk 里加了口语化注释，比如'硅胶隔膜（也叫硅胶片）'，解决用户用词和文档用词不一致的问题。
>
> 优化后 Hit Rate@3 从 72% 提升到 92%。"

### "如何证明模型效果变好了？"

> "我们有 25 个测试用例，每个标注了期望关键词和期望命中 chunk。运行评测计算 Hit Rate@3、MRR、关键词覆盖率三个指标。
>
> V1 到 V2 只改了 chunk 切分方式，其他不变，用同一套测试集对比。Hit Rate@3 从 72% 到 92%，MRR 从 0.553 到 0.790，证明优化有效。"

### "怎么保证工作流的稳定性？"

> "三道防线：规则过滤闲聊、相关性阈值检查（低于阈值触发降级）、答案自检（关键词覆盖率检查）。
>
> 宁可说'我不知道'，也不能编造答案，尤其是母婴产品。"

### "Chunk 切分有什么讲究？"

> "一个 chunk 一个独立知识点。安全警告每条独立，用户问'充电时能用吗'直接命中对应那条。
>
> 专业名词加口语化注释，比如'拆解（拆、拆卸、取下）'，让 BM25 和向量都能匹配上。这不是一次定死的，上线后根据用户实际问题持续迭代。"
