# SporraMom W3 智能客服系统 — 优化过程与效果记录

## 一、项目概述

为 SporraMom W3 穿戴式吸奶器构建的 RAG 智能客服系统。本文档记录了从 V1 到 V2 的完整优化过程，包括发现的问题、优化方案、以及量化效果对比。

---

## 二、V1 版本：自动切分方案

### 2.1 切分方式

使用 Python 代码自动解析 Markdown 标题层级，按 `#` `##` `###` 切分。

### 2.2 发现的问题

| 问题 | 具体表现 | 影响 |
|------|---------|------|
| **chunk 太短** | Chunk 1 只有 21 字（产品名），Chunk 6 只有 79 字 | 向量编码质量差，语义信息不足 |
| **无用 chunk** | Chunk 5 是目录，对回答问题没有帮助 | 浪费检索资源，干扰排序 |
| **长 chunk 没切** | 安全警告 1-12 条混在 2 个大 chunk 里（769字+548字） | 用户问单条警告时，检索到的是整个大块，LLM 难以定位 |
| **chunk 重叠** | Chunk 3 和 4 的第 8-9 条安全警告重复 | 重复信息干扰排序 |
| **粒度不均** | 最短 21 字，最长 769 字，差距 36 倍 | 检索效果不稳定 |
| **缺少口语化表达** | chunk 里只有书面语（"分离主机"），没有口语（"怎么拆"） | 用户说口语时检索分数低 |

### 2.3 V1 评测结果

```
测试用例: 25 个
Hit Rate@3:    72%    （25 个用例中 18 个在 top-3 命中正确 chunk）
MRR:           0.553  （正确 chunk 平均排在第 1.8 名）
关键词覆盖率:  81.4%  （答案中 81.4% 的期望关键词被包含）
降级触发率:    8%     （2 个用例触发了"答不上来"的降级策略）
自检通过率:    100%
平均响应时间:  17.6s
```

## 三、优化过程

### 3.1 流程优化

**V1 流程（3 次 LLM 调用）：**
```
用户提问 → LLM 意图识别（4分类）→ 检索 → LLM 生成答案 → LLM 自检
```

**V2 流程（1 次 LLM 调用）：**
```
用户提问 → 规则过滤闲聊 → 检索 → 相关性检查（看分数）→ LLM 生成答案 → 规则自检
```

| 优化点 | V1 | V2 | 原因 |
|--------|----|----|------|
| 意图识别 | LLM 判断 4 类 | 规则判断闲聊/产品问题 | LLM 调用慢，且意图判断不需要那么复杂 |
| 超范围判断 | LLM 猜 | 检索分数决定 | 知识库里没有 = 超范围，比 LLM 猜更可靠 |
| LLM 调用次数 | 3 次 | 1 次 | 响应更快，成本更低 |

### 3.2 Chunk 切分优化

**核心原则：一个 chunk = 一个独立的知识点**

#### 安全警告：从 2 大块 → 12 个小块

```
V1: 2 个 chunk（769字 + 548字），12 条警告混在一起，有重叠
V2: 12 个 chunk，每条警告独立，每条都加了"温馨提示"
```

**效果**：用户问"充电时能用吗"，直接命中"安全警告-充电时禁止使用"，不用从 769 字里找。

#### 按钮和 LED：从 2 个 → 4 个（按功能分组）

```
V1: 按钮 1 个 chunk + LED 1 个 chunk
V2: 电源键与切换键 / 档位调节与释放按钮 / LED 电量与充电 / LED 模式与档位
```

**效果**：用户问"怎么开机"，命中"电源键与切换键操作"，不会混入 LED 信息。

#### 尺寸选择：从 1 个 → 2 个

```
V1: 选择 + 测量混在一个 chunk
V2: "尺寸选择与不当后果" + "尺寸测量方法"
```

**效果**：用户问"尺寸选错会怎样"只命中选择篇，问"怎么测量"只命中测量篇。

#### 充电时间与续航：合并为 1 个

```
V1: 充电信息散落在多个 chunk
V2: 合并为"充电时间与续航"（充电时间、续航、电池容量、注意事项）
```

**效果**：用户问"电池能用多久"，一个 chunk 就能完整回答。

#### 专业名词加口语化注释

```
V1: "从主机底部密封槽内轻轻剥离硅胶隔膜"
V2: "拆下硅胶隔膜（也叫硅胶片），从主机底部的密封槽里轻轻剥下来"
```

**加了同义词的 chunk：**
- 硅胶隔膜 = 硅胶片
- 喇叭罩 = 吸乳罩 = 护罩
- 集乳杯 = 奶杯
- 拆解 = 拆卸 = 拆 = 取下 = 卸下

**效果**：用户说"硅胶片怎么装"也能命中，不会因为用词不同而检索不到。

### 3.3 为什么加同义词有用？

**问题**：向量检索能理解近义词，但分数不够高。

```
用户问: "怎么拆"
chunk 里写: "分离主机"（没有"拆"字）

BM25: "拆" 在 chunk 里没出现 → 分数 0
向量: "拆" 和 "分离" 语义接近 → 分数 0.78（不是最高）

→ 能找到，但排名靠后
```

```
加了同义词后:
chunk 写: "怎么拆（拆解、拆卸、取下）：把主机分离"

BM25: "拆" 在 chunk 里出现了 → 分数高
向量: "拆" 和 "拆" 完全匹配 → 分数 1.0

→ 能找到，而且排名靠前
```

**一句话：加同义词不是因为向量检索没用，而是让它分数更高、排名更靠前。**

## 五、指标计算方法

### 5.1 Hit Rate@K

```
含义: top-K 个检索结果中，是否包含"正确答案"所在的 chunk

计算:
  25 个测试用例，每个用例检索后看 top-3
  命中 → 1，未命中 → 0
  Hit Rate@3 = 命中数 / 25

示例:
  问题: "吸奶器可以在充电的时候用吗？"
  期望: "安全警告-充电时禁止使用"
  top-3: [安全警告-充电时禁止使用, 安全警告-仅限哺乳期使用, 充电时间与续航]
  第 1 个就命中 → Hit Rate@3 = 1
```

### 5.2 MRR（Mean Reciprocal Rank）

```
含义: 正确 chunk 排在第几名的倒数，衡量"排得够不够靠前"

公式: MRR = (1/排名1 + 1/排名2 + ... + 1/排名N) / N

示例:
  问题1: 正确 chunk 排第1名 → 1/1 = 1.0
  问题2: 正确 chunk 排第2名 → 1/2 = 0.5
  问题3: 正确 chunk 排第5名 → 1/5 = 0.2
  问题4: 没找到           → 0
  
  MRR = (1.0 + 0.5 + 0.2 + 0) / 4 = 0.425
```

### 5.3 关键词覆盖率

```
含义: 生成的答案中，包含了多少"期望关键词"

公式: 覆盖率 = 答案中出现的期望关键词数 / 期望关键词总数

示例:
  问题: "吸奶器可以在充电的时候用吗？"
  期望关键词: ["充电", "禁止", "边充边用", "电气安全"]
  答案: "充电时禁止佩戴使用，避免电气安全风险"
  包含: "充电" ✅ "禁止" ✅ "边充边用" ❌ "电气安全" ✅
  覆盖率 = 3/4 = 75%
```

### 5.4 降级触发率

```
含义: 有多少用例触发了"答不上来"的降级策略

计算: 检索最高分 < 阈值 → 触发降级
      降级触发率 = 触发降级的用例数 / 25
```

### 5.5 自检通过率

```
含义: 生成的答案是否通过了"忠实度检查"

计算: 关键词覆盖率 >= 30% → 通过
      自检通过率 = 通过的用例数 / 25
```

## 六、技术架构

```
用户提问
    ↓
规则过滤（"你好/谢谢" → 闲聊回复）
    ↓
混合检索（BM25 关键词 + 向量语义 + RRF 融合 + Rerank 精排）
    ↓
相关性检查（rerank 分数 >= 阈值？）
    ↓ 是 → 正常回答
    ↓ 否 → 低置信度提示 / 转人工客服
LLM 生成答案（读取检索到的 chunk，组织语言回答）
    ↓
规则自检（关键词覆盖率 >= 30%？）
    ↓ 是 → 返回答案
    ↓ 否 → 低温度重试
最终答案
```

### 关键技术选型

| 组件 | 选型 | 选择理由 |
|------|------|----------|
| Embedding | bge-small-zh-v1.5 | 中文优化，90MB，GPU 友好 |
| Reranker | bge-reranker-base | CrossEncoder，比双塔更精准 |
| BM25 | rank_bm25 + jieba | 纯 Python，无需训练 |
| 向量库 | FAISS | 本地部署，无需外部服务 |
| LLM | mimo-v2.5-pro | 中文能力强，API 兼容 OpenAI |
| 前端 | Gradio | 快速搭建，内置调试工具 |

## 八、面试话术

### "你们是怎么做数据处理的？"

> "我们的数据源是产品说明书，一份 Markdown 文档。最初用代码按标题自动切分，但效果不好——chunk 粒度不均，有的 20 字有的 700 字，安全警告 12 条混在 2 个大块里。
>
> 后来改成人工按逻辑单元切分，每个 chunk 是一个独立的知识点。比如安全警告每条一个 chunk，故障排除每个类型一个 chunk。同时在 chunk 里加了口语化注释，比如'硅胶隔膜（也叫硅胶片）'，解决用户用词和文档用词不一致的问题。
>
> 优化后 Hit Rate@3 从 72% 提升到 92%，MRR 从 0.553 提升到 0.790。"

### "如何证明模型效果变好了？"

> "我们有一套评测体系，包含 25 个测试用例，覆盖安全、操作、故障、参数等 6 大类场景。
>
> 每个用例有三个标注：用户问题、期望关键词、期望命中 chunk。运行评测后计算 5 个指标：Hit Rate@3（找没找对）、MRR（排得够不够前）、关键词覆盖率（答全没有）、降级触发率（答不上来的比例）、自检通过率（有没有编造信息）。
>
> V1 到 V2 的优化，我们只改了 chunk 切分方式，其他都不变，用同一套测试集对比，Hit Rate@3 从 72% 提升到 92%，证明优化有效。"

### "怎么保证工作流的稳定性？"

> "我们设计了三道防线：
>
> 第一道，规则过滤闲聊，避免无关问题走检索流程。
>
> 第二道，相关性阈值检查。检索结果的 rerank 分数低于阈值，说明知识库里没有相关内容，触发降级策略——先扩大检索范围，仍不足就提示用户联系客服。宁可说'我不知道'，也不能编造答案，尤其是吸奶器这种母婴产品。
>
> 第三道，答案自检。用关键词覆盖率检查答案是否忠实于源文档，覆盖率低就用更低温度重新生成。"

### "Chunk 切分有什么讲究？"

> "核心原则是一个 chunk 一个独立知识点。具体来说：
>
> 安全警告每条独立——用户问'充电时能用吗'，直接命中对应那条，不用从 700 字里找。
>
> 专业名词加口语化注释——用户说'怎么拆'，chunk 里写'拆解（拆卸、取下）'，BM25 和向量都能匹配上。
>
> 这不是一次定死的，是持续迭代的。上线后看用户实际问了什么搜不到，就把那个词加到 chunk 里。"
