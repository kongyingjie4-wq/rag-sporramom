# SporraMom W3 RAG 客服系统 — 原理讲解

## 一、这个系统到底在干什么？

### 一句话解释

用户问一个问题，系统从产品说明书里找到相关内容，然后用 AI 基于这些内容写出回答。

### 用生活类比

想象你是一个客服，桌上放着一本产品说明书。

1. 客户问你："吸奶器充不了电怎么办？"
2. 你翻说明书，找到"故障排除"那一章
3. 你读了相关段落
4. 你用自己的话组织语言，把答案告诉客户

**RAG 系统做的事和你一模一样**，只不过"翻说明书"这一步用的是计算机技术。

## 二、核心概念解释

### 2.1 RAG 是什么？

**RAG** = **R**etrieval **A**ugmented **G**eneration

拆开看：
- **Retrieval**（检索）= 从文档库中"找到"相关内容 → 相当于"翻说明书"
- **Augmented**（增强）= 用找到的内容来"增强"AI的回答 → 相当于"先读资料再回答"
- **Generation**（生成）= AI 用自己的话写出最终答案 → 相当于"组织语言"

**为什么不直接问 AI？**

因为 AI（大语言模型）的知识是训练时学到的，它不知道 SporraMom W3 这个具体产品的信息。就像一个新来的客服，没读过说明书，你问他他也答不上来。

RAG 的做法是：**先帮他翻说明书，把相关段落指给他看，让他基于这些段落来回答**。

### 2.2 什么是"向量"（Vector）？

**向量就是一串数字，用来表示一段文字的"含义"。**

举个例子，假设我们用 5 个数字来表示文字的含义：

```
"吸奶器充不了电" → [0.8, 0.1, 0.9, 0.2, 0.3]
"无法充电怎么办" → [0.7, 0.1, 0.8, 0.3, 0.3]
"母乳保存时间"   → [0.1, 0.9, 0.1, 0.8, 0.2]
```

你会发现：
- "充不了电"和"无法充电"的数字很像 → 因为它们**意思相近**
- "充不了电"和"母乳保存"的数字很不同 → 因为它们**意思不同**

**计算机通过比较这些数字，就能判断两段话是不是在说同一件事。**

这个过程叫 **Embedding**（嵌入），把文字变成数字向量。

### 2.3 什么是"余弦相似度"（Cosine Similarity）？

**用来衡量两个向量有多"像"的数学方法。**

- 结果范围：-1 到 1
- **1** = 完全一样方向 = 意思完全相同
- **0** = 垂直 = 完全无关
- **-1** = 完全相反

类比：两个人站在原点，分别朝一个方向走。余弦相似度就是看他们走的方向有多接近。

```
"充不了电" 和 "无法充电"  → 相似度 0.95（方向几乎一样）
"充不了电" 和 "母乳保存"  → 相似度 0.12（方向完全不同）
```

### 2.4 什么是 BM25？

**BM25 是一种"关键词匹配"算法。**

它做的事很简单：看用户的问题里有哪些词，然后在文档里找包含这些词最多的段落。

```
用户问题: "吸奶器 充不了 电"
         ↓ 分词
关键词: [吸奶器, 充不了, 电]

文档1: "吸奶器无法充电..." → 包含"吸奶器"和"电" → 分数高
文档2: "母乳保存方法..."   → 包含"电"（电池）  → 分数低
```

**BM25 的优势**: 精确匹配关键词，比如型号"YM-8871"、参数"1400mAh"
**BM25 的劣势**: 理解不了同义词，"充不了电"和"无法充电"对它来说是不同的词

### 2.5 什么是 FAISS？

**FAISS 是 Facebook 开发的向量检索工具。**

它的作用：给定一个 query 向量，快速从几百万个向量中找到最相似的 top-K 个。

类比：你有一个巨大的电话本，你想找和你家地址最近的 5 个人。FAISS 就是帮你快速算出谁离你最近的工具。

在我们的系统里：
- 我们把 23 个 chunk 都编码成向量，存进 FAISS
- 用户提问时，把问题也编码成向量
- FAISS 帮我们找到最相似的 5-10 个 chunk

### 2.6 什么是 RRF（Reciprocal Rank Fusion）？

**RRF 是一种"融合两个排序"的方法。**

我们有两种检索方式（BM25 和 向量），它们各自给出一个排序。问题是：怎么把两个排序合并成一个？

**为什么不能直接比分数？**
- BM25 的分数可能是 0-10
- 向量相似度的分数是 0-1
- 两个分数的"单位"不同，没法直接比较

**RRF 的做法：只看排名，不看分数。**

公式：`score(doc) = 1/(60 + BM25排名) + 1/(60 + 向量排名)`

举个例子：

| 文档 | BM25排名 | 向量排名 | RRF分数 |
|------|---------|---------|---------|
| 文档A | 第1名 | 第3名 | 1/61 + 1/63 = 0.0323 |
| 文档B | 第5名 | 第1名 | 1/65 + 1/61 = 0.0317 |
| 文档C | 第2名 | 第2名 | 1/62 + 1/62 = 0.0323 |

**结果**: 文档A和C并列最高（两种检索都认为它们不错），文档B稍低。

**RRF 的优势**: 简单、鲁棒，不依赖分数绝对值。

### 2.7 什么是 Reranker（重排序器）？

**Reranker 是一种更精准的"打分"模型，用来对候选结果做最终排序。**

前面的 BM25 和 向量检索都是"粗筛"——快速从 23 个 chunk 中选出 top-10 候选。

但粗筛不够精准。Reranker 做的事是：**把用户问题和候选文档拼在一起，让模型仔细阅读，给出一个精准的相关性分数。**

```
粗筛（Embedding 双塔模型）:
  问题 → 编码成向量A
  文档 → 编码成向量B
  相似度 = 向量A 和 向量B 的余弦相似度
  （问题和文档各自独立编码，没有交互）

精排（Reranker CrossEncoder）:
  [问题] + [文档] → 拼接成一个输入 → 模型输出相关性分数
  （问题和文档一起输入，模型可以看到它们之间的关系）
```

**类比**：
- 粗筛 = 你快速翻目录，看哪些章节标题可能相关
- 精排 = 你仔细阅读这些章节的内容，判断哪个最相关

**为什么不在所有文档上都用 Reranker？**
因为 Reranker 要把问题和文档拼在一起输入，计算量大。23 个 chunk 可以接受，但如果有 10 万个 chunk 就太慢了。所以先用粗筛选出 top-10，再用精排。

### 2.8 什么是 CrossEncoder？

**CrossEncoder 就是 Reranker 使用的模型架构。**

- **双塔模型（Bi-Encoder）**: 问题和文档分别编码成向量，然后比较。快但不够准。
- **交叉编码器（CrossEncoder）**: 问题和文档拼在一起输入，模型内部有"交叉注意力"，能看到词与词之间的关系。准但更慢。

```
双塔: [问题] → 向量A ─┐
                       ├→ 比较
       [文档] → 向量B ─┘

交叉: [问题 + 文档] → 一个模型 → 相关性分数
```

### 2.9 什么是 Temperature（温度）？

**控制 AI 回答"确定性"的参数。**

- **Temperature = 0**: AI 总是选最可能的下一个词 → 回答最确定、最忠实于原文
- **Temperature = 0.3**: 稍微有点随机 → 回答基本确定，偶尔有小变化
- **Temperature = 1.0**: 比较随机 → 回答更多样，但可能偏离原文

**在我们的系统里**：
- 正常回答用 temperature=0.3（基本忠实，稍有灵活性）
- 自检失败后重试用 temperature=0.0（最大程度忠实于原文）

### 2.10 什么是 Token？

**Token 是 AI 处理文字的最小单位。**

- 英文：一个单词或一个词根通常是一个 token
- 中文：一个字或一个词通常是一个 token

```
"吸奶器充不了电" → 可能被分成 4-5 个 token
```

**为什么关心 token 数？**
- API 按 token 计费
- 模型有 token 上限（输入+输出不能超过某个数）
- token 越多，处理越慢

## 四、前端界面每个 Tab 看什么

### Tab 1: 🔬 全链路调试

输入一个问题，展开每个 Step，你能看到：

| Step | 你看什么 | 有什么用 |
|------|---------|---------|
| Step 1 | 意图类别 | 判断系统有没有理解用户在问什么 |
| Step 2 | BM25 分词和分数 | 看关键词匹配准不准 |
| Step 3 | 向量和余弦相似度 | 看语义理解准不准 |
| Step 4 | RRF 融合排名 | 看两种检索结果怎么合并的 |
| Step 5 | Rerank 打分 | 看精排后哪个文档排第一 |
| Step 6 | 阈值检查 | 看有没有触发降级策略 |
| Step 7 | 完整 Prompt | 看发给 LLM 的原始输入 |
| Step 8 | LLM 回答 | 看 AI 生成的最终答案 |
| Step 9 | 自检结果 | 看答案是否忠实于原文 |

### Tab 2: 📦 向量数据库

- **加载全部 Chunk**: 看系统把产品说明书切成了多少块，每块的内容是什么
- **向量搜索**: 输入任意文本，看它和每个 chunk 的相似度排序

**你能在这里发现的问题**：
- 某个 chunk 切得不好（信息被切断了）
- 某个查询应该匹配 A 但实际匹配了 B（检索质量差）

### Tab 3: 📊 评测

一键运行 25 个测试用例，看系统整体表现。

|------|------|-----------|
| **RAG** | 检索增强生成 | 先找资料，再让 AI 回答 |
